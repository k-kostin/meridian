# ROADMAP.md

Практическая дорожная карта развития проекта после текущего MVP.

Последнее обновление: 2026-06-03

## 0. Версионная рамка

- Текущий рабочий MVP зафиксирован как `v0.1 MVP Baseline` в `docs/releases/v0.1-mvp-baseline.md`.
- Новая функциональность из Twenty-inspired плана начинается с `v0.2`.
- Каждая следующая версия должна сохранять do-not-regress список из baseline release note.

## 1. Текущее положение

Сейчас проект находится в рабочем состоянии для:

- локальной разработки;
- demo/pilot показа;
- малого внутреннего контура;
- базового складского учета с расчетными остатками, инвентаризацией и Excel-выгрузками.

На текущем этапе стек и архитектура оправданы:

- Django server-rendered UI;
- SQLite как стартовая БД;
- один Django app;
- аналитика и Excel-экспорт в коде приложения;
- demo mode для показа продукта.

Это хороший MVP, но не финальная операционная конфигурация.

## 2. Ближайшая дорожная карта

### Этап A. Стабилизация MVP

Цель: сделать контур безопасным и понятным для первых реальных пользователей.

Приоритет:

1. Workflow hardening:
   - роли и базовое разграничение доступа;
   - запрет опасных действий для проведенных документов во всех каналах;
   - проверка audit trail.
2. UX hardening:
   - итоговые строки в отчетах там, где они реально помогают;
   - улучшение читаемости Excel-выгрузок;
   - финальная шлифовка onboarding и demo flow.

Что уже закрыто в текущем коде:

- coarse-grained concurrency hardening для нумерации и проведения документов;
- отделение Excel workbook services от HTTP-ответов.

### Этап B. Операционный контур

Цель: сделать систему пригодной для малого рабочего использования.

Приоритет:

1. Редактирование черновиков через web-экраны.
2. Импорт номенклатуры и стартовых остатков из Excel.
3. Документ перемещения между складами, если он нужен бизнесу.
4. Категории номенклатуры и более удобные фильтры отчетов.

### Этап C. Аналитика поверх MVP

Цель: добавить полезную аналитику без превращения системы в тяжелый BI-комбайн.

Приоритет:

1. Итоги по складам и общий итог в периодной аналитике.
2. Дополнительные агрегаты:
   - по складам;
   - по группам номенклатуры;
   - по типам движения.
3. Простые графики по периоду:
   - динамика прихода/расхода;
   - структура остатков;
   - top positions by movement.

### Этап D. Плановый контур и прогнозирование

Цель: добавить слой операционного планирования поверх фактического учета.

Идея:

- вести календарь ожидаемых поставок;
- вести календарь планируемых отгрузок / продаж;
- хранить не только факт, но и будущие обязательства по датам;
- на этой базе строить прогноз доступного остатка по складу и позиции.

Практический смысл:

- видеть ожидаемый дефицит заранее;
- понимать, в какие даты склад уйдет ниже безопасного уровня;
- оценивать влияние будущих поставок и отгрузок на доступность номенклатуры;
- готовить простой операционный forecast без полноценной ERP/MRP.

Минимальный возможный сценарий:

1. Отдельный календарь плановых событий:
   - `ожидаемое поступление`;
   - `плановая отгрузка`.
2. У каждого события есть:
   - дата;
   - склад;
   - позиция;
   - количество;
   - статус (`план`, `подтверждено`, `отменено`, `исполнено`).
3. Прогноз остатка считается как:
   - текущий расчетный остаток;
   - плюс будущие подтвержденные поступления;
   - минус будущие подтвержденные отгрузки;
   - с просмотром по дням/неделям на горизонте.

Важно:

- это отдельный будущий слой, а не часть текущего MVP;
- прогноз не должен подменять фактический учет;
- сначала должен оставаться один источник истины для факта, потом поверх него — плановый календарь и forecast.

### Этап E. Desktop / GUI App

Цель: упаковать текущий контур в Windows-приложение без внешнего браузера и без ручной установки Python у пользователя.

Выбранный подход:

1. Primary path — `Electron + Python sidecar`.
2. Experimental path — `Tauri + тот же Python sidecar`.
3. Fast fallback — `pywebview + тот же Python sidecar`.

Почему именно так:

- сохраняется текущий Django UI;
- не требуется отдельный UI rewrite;
- Electron убирает runtime-зависимость от WebView2 и делает первый запуск на Windows предсказуемее;
- Tauri остается экспериментом на случай, если размер дистрибутива станет важнее.

Минимальный план:

1. Поднять локальный sidecar на `waitress`.
2. Завести Electron shell вокруг sidecar.
3. Подготовить NSIS per-user installer.
4. Проверить установку без admin-прав, Excel-выгрузки и SQLite data-dir.
5. Проверить Tauri отдельно как эксперимент, не блокирующий Electron-релиз.

## 3. SQLite и масштабирование до нескольких пользователей

### Финальная позиция

SQLite подходит для:

- разработки;
- demo mode;
- пилота;
- малого внутреннего контура с невысокой интенсивностью записи.

SQLite, скорее всего, будет приемлем и для 3-5 пользователей, если:

- пользователи в основном смотрят данные;
- записи происходят не непрерывно, а эпизодически;
- нет постоянного одновременного проведения большого числа документов.

SQLite начнет мешать, если:

- несколько операторов регулярно одновременно проводят документы;
- появляются задержки или `database is locked`;
- растет число параллельных write-операций;
- нужна более предсказуемая серверная эксплуатация и резервирование.

### Практический вывод

- Для первого реального запуска `SQLite` можно оставить.
- Для устойчивой многопользовательской эксплуатации стоит планировать переход на `PostgreSQL`.
- Переход желательно делать до того, как система станет критичной для ежедневной работы.

### Почему переход на PostgreSQL здесь не должен быть болезненным

Проект построен на Django ORM, поэтому основная доменная логика не зависит от конкретной БД.

Нормальный путь миграции:

1. Поднять `PostgreSQL`.
2. Настроить `DATABASES` в Django settings.
3. Прогнать `migrate`.
4. Перенести данные.
5. Переключить приложение на новую БД.

То есть архитектурной ловушки сейчас нет, но есть технический долг по конкурентным операциям, который все равно нужно закрывать независимо от БД.

### Рекомендуемая политика

- Пока проект в стадии MVP/pilot: `SQLite`.
- Перед активной многопользовательской эксплуатацией: `PostgreSQL`.
- Для BI и внешних dashboards: почти наверняка `PostgreSQL`.

## 4. Встроенная аналитика через pandas / polars

### Что здесь имеет смысл

Добавить модуль "легкой аналитики", который:

- берет те же фильтры периода, склада и представления, что уже есть в UI;
- считает дополнительные агрегаты поверх выгрузки/набора строк;
- показывает результат прямо в браузере;
- не ломает текущий UX и не создает отдельный параллельный продукт внутри системы.

### Что лучше считать через ORM, а что через pandas/polars

Через ORM удобно:

- текущие остатки;
- базовые period reports;
- фильтрацию по складу, дате, статусу;
- простые агрегаты, близкие к доменной логике.

Через `pandas` или `polars` удобно:

- pivot / cross-tab;
- группировки по нескольким измерениям;
- доли, ranking, top-N;
- более сложные табличные summary и экспериментальную аналитику.

### Выбор между pandas и polars

`pandas`:

- привычнее;
- проще для большинства разработчиков;
- лучше подходит для первого аналитического модуля.

`polars`:

- быстрее на больших таблицах;
- очень удобен для columnar-style агрегаций;
- хорош, если аналитических сценариев станет заметно больше.

### Практическая рекомендация

- Для первого встроенного аналитического модуля начать с `pandas`, если нужен быстрый и предсказуемый результат.
- Если аналитика начнет расти по объему и сложности, можно перейти на `polars` либо использовать его точечно.

## 5. Как показывать такую аналитику в браузере

### Нормальный UX-подход

Не делать отдельный "аналитический кабинет" с другой логикой фильтров.

Правильнее:

1. Оставить существующие фильтры периода/склада.
2. Добавить вкладку или блок "Базовая аналитика" рядом с текущим периодным отчетом.
3. Показывать:
   - summary cards;
   - 1-3 полезных графика;
   - компактные вспомогательные таблицы.

### Что не стоит делать для product UI

- Не делать основной веб-UI на `seaborn` картинках.
- Не тащить тяжелые notebook-style визуализации в production flow.
- Не строить много графиков без реальной операционной пользы.

### Что стоит делать

- backend считает агрегаты;
- frontend получает компактный JSON;
- графики рисуются в браузере.

Рекомендуемые варианты:

- `Plotly` — если нужна интерактивность;
- `Altair` — если нужна аккуратная декларативная визуализация;
- `Chart.js` — если нужны простые и легкие product-графики.

### Практическая рекомендация

Для этого проекта самый прагматичный путь:

- табличные расчеты в Django + ORM или `pandas/polars`;
- графики в браузере через `Chart.js` или `Plotly`;
- единые фильтры и единый экран, без UX-разрыва.

## 6. Superset

### Можно ли тянуть данные в Superset

Технически да, но как рабочую траекторию лучше рассматривать не `SQLite`, а `PostgreSQL`.

Причина проста:

- Superset естественнее работает с серверной БД;
- проще давать read-only доступ;
- проще строить SQL views и dashboards;
- проще эксплуатация и масштабирование.

### Рекомендуемый путь

1. Перевести основную БД на `PostgreSQL`.
2. Подключить Superset read-only пользователем.
3. Для BI-потребностей сделать отдельные представления или curated views:
   - движения за период;
   - остатки по складам;
   - инвентаризационные отклонения;
   - агрегаты по группам/единицам/типам движения.
4. Использовать Superset для manager-facing dashboards, а не для операционного ввода документов.

### Практический вывод

- Superset сюда хорошо ложится как внешний BI-слой.
- Но правильный момент для него — после перехода на `PostgreSQL`.
- Внутренний UI и Superset должны дополнять друг друга, а не конкурировать:
  - WMS UI = операции и базовая аналитика;
  - Superset = дашборды, сравнения, BI-срезы.

## 7. Рекомендуемая целевая траектория

Если идти прагматично, без перегруза MVP:

1. Дожать стабильность текущего контура.
2. Закрыть конкурентные риски.
3. Решить вопрос ролей и черновиков.
4. При первых признаках реальной многопользовательской работы перейти на `PostgreSQL`.
5. Добавить легкую встроенную аналитику в браузере.
6. После этого подключать Superset как внешний слой управленческой аналитики.

## 8. Финальные мысли

- `SQLite` сейчас не ошибка, а нормальный выбор для стадии продукта.
- `PostgreSQL` нужен не “потому что так принято”, а когда появляется многопользовательская эксплуатация и требования к устойчивости.
- Встроенная аналитика через `pandas/polars` вполне совместима с текущим UX, если не делать из нее отдельную BI-систему.
- `Superset` сюда хорошо вписывается, но как следующий слой зрелости, а не как первый шаг.
- Для этого проекта лучше эволюционный путь: сначала стабильный MVP, затем многопользовательская надежность, затем расширенная аналитика.
