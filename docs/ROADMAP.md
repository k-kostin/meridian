## 0. Версионная рамка

- Текущий рабочий MVP зафиксирован как `v0.1 MVP Baseline` в `docs/releases/v0.1-mvp-baseline.md`.
- Новая функциональность начинается с `v0.2`; первый foundation slice фиксирует версионность приложения.
- Каждая следующая версия должна сохранять do-not-regress список из baseline release note.
- Версия приложения не повышается на каждый pull request. Patch/slice release notes могут фиксировать отдельные срезы, но UI/package version повышается только на понятных milestone-точках.

## 1. Текущее положение

Сейчас проект находится в рабочем состоянии для:

- локальной разработки;
- demo/pilot показа;
- малого внутреннего контура;
- базового складского учета с расчетными остатками, инвентаризацией и Excel-выгрузками.

Продуктовая рамка после external product review:

- Meridian не должен позиционироваться как новая ERP или полноценная WMS;
- сильная ниша - маленький локальный складской desk для микробизнеса, который перерос Excel;
- целевой ранний контур - один компьютер, один активный оператор, владелец/наблюдатель и Excel-first onboarding;
- конкурировать нужно не полнотой ERP-функций, а простотой установки, безопасным переносом Excel-данных, понятной складской логикой и низким порогом внедрения.

Коммерческая модель поставки:

- один доменный core и одна реализация складских правил для всех вариантов;
- `Local Single User` - основной ранний коммерческий профиль: SQLite, один Windows-компьютер, один активный оператор, локальная база, backup/restore и понятная установка;
- `Team / Multi-User` - будущий расширенный профиль: тот же Django/core, но server/PostgreSQL deployment, несколько рабочих мест, user attribution, backup policy и проверенная конкурентная эксплуатация;
- не заводить отдельные SQLite- и PostgreSQL-версии бизнес-логики. Различия должны жить в settings, packaging, deployment, data-dir, backup/update процедурах и операционных гарантиях.

На текущем этапе стек и архитектура оправданы:

- Django server-rendered UI;
- SQLite как стартовая БД;
- один Django app;
- аналитика и Excel-экспорт в коде приложения;
- demo mode для показа продукта.

Это хороший MVP, но не финальная операционная конфигурация. До реального коммерческого пилота важнее закрыть data-safety, attribution и installer readiness, чем добавлять более богатую аналитику.

## 2. Ближайшая дорожная карта

### Этап A. Стабилизация MVP

Цель: сделать контур безопасным и понятным для первых реальных пользователей.

Приоритет:

1. Workflow hardening:
   - роли и базовое разграничение доступа;
   - запрет опасных действий для проведенных документов во всех каналах;
   - операционная история событий для складо-влияющих действий.
2. UX hardening:
   - итоговые строки в отчетах там, где они реально помогают;
   - улучшение читаемости Excel-выгрузок;
   - финальная шлифовка onboarding и demo flow.

Что уже закрыто в текущем коде:

- coarse-grained concurrency hardening для нумерации и проведения документов;
- отделение Excel workbook services от HTTP-ответов.
- операционная история событий для проведения документов и инвентаризаций.
- встроенные быстрые фильтры для журнала документов движения и текущих остатков.
- базовая role-aware UX-модель `admin / operator / viewer` для пилотного использования.
- parser-only foundation для будущего импорта Excel-номенклатуры.
- UI предпросмотра импорта Excel-номенклатуры без записи изменений.
- commit-flow для создания новых позиций номенклатуры из валидного Excel-файла.
- import-flow стартовых остатков из Excel через создание черновика полной инвентаризации.

Stage A audit gate passed on 2026-06-04; new product work should continue from Stage B.

### Этап B. Операционный контур

Цель: сделать систему пригодной для малого рабочего использования.

Статус: закрыт как `v0.4.0`.

Что закрыто:

- Stage B scope reconciliation без переоткрытия уже реализованных черновиков, перемещений и встроенных presets.
- Расширение импорта номенклатуры:
  - распространенные алиасы колонок;
  - явный update mode для существующих позиций;
  - явный auto-create mode для недостающих единиц измерения.
- Категории номенклатуры и фильтр по категории в справочнике.
- Category-aware фильтры в остатках, `Дни`, `Месяцы`, аналитике периода и соответствующих Excel-выгрузках.
- Пользовательские сохраненные представления для документов движения и текущих остатков.

Что не входило в Stage B и остается будущей работой:

- production-grade RBAC, object-level permissions, приглашения и восстановление пароля;
- расширенный импорт стартовых остатков для нескольких складов в одном файле;
- контрагенты, цены и связь со сбытом;
- default view, sharing и права доступа для сохраненных представлений.

### Этап C. Commercial Pilot Readiness

Цель: сделать текущий маленький складской desk безопасным для первого реального клиента, а не просто функционально интересным demo.

Почему этот этап идет перед аналитикой:

- для микробизнеса потеря локальной базы важнее отсутствия красивого графика;
- владелец быстро спросит, кто создал или провел операцию;
- локальный desktop-продукт обязан иметь понятный data-dir, backup/restore и обновление без ручного DevOps;
- SQLite подходит для локального single-active-operator контура, но это ограничение нужно явно держать в продуктовой рамке.

Приоритет:

1. Data safety:
   - явный экран/команда ручного backup;
   - restore procedure;
   - автоматический backup перед миграцией/обновлением;
   - понятное расположение пользовательской SQLite-базы.

First slice implemented: local manual backup, restore command, and pre-migration backup for SQLite. Scheduled/cloud/encrypted backups remain outside this slice.

2. User attribution:
   - кто создал документ;
   - кто изменил черновик;
   - кто провел документ или инвентаризацию;
   - кто перезагрузил demo/import data, если действие доступно.

First slice implemented: creator/updater/poster fields for stock documents and inventory documents, plus actor labels on posting timeline events.

Audit hardening slice implemented: `ActivityEvent` now records successful item import commits, opening inventory import commits, manual backup creation, demo data load/reset and reference record changes. This is a pilot-grade operational trail, not an immutable or security-grade audit log.

3. Audit hardening:
   - текущий `ActivityEvent` расширить до operational audit trail для пилота;
   - не обещать security-grade audit log без отдельного проекта.
4. Deployment limits:
   - явно зафиксировать supported mode: один локальный компьютер и один активный оператор;
   - не обещать полноценную одновременную многопользовательскую работу на SQLite;
   - для реального multiuser path отдельно планировать PostgreSQL/server deployment;
   - оформить границу между `Local Single User` и будущим `Team / Multi-User` профилем без развилки доменной логики.

Deployment limits slice implemented: dashboard and docs explicitly state the supported Local Single User profile: SQLite, one local computer, one active operator. Team / Multi-User remains a future server/PostgreSQL profile using the same domain core.

5. Real Excel onboarding validation:
   - проверить импорт на 2-3 реальных клиентских или приближенных к реальным Excel-структурах;
   - улучшать alias/mapping только по фактическим файлам, а не по воображаемым ERP-сценариям.

Status: intentionally deferred until after the Stage C audit/deployment gate and a separate user command.

Не включать в этот этап:

- кассу, продажи, маркетплейсы, бухгалтерию;
- production-grade RBAC;
- object-level permissions;
- barcode/labels без подтвержденного клиентского сценария.

### Этап D. Desktop / GUI App

Цель: упаковать текущий контур в Windows-приложение без внешнего браузера и без ручной установки Python у пользователя.

Выбранный подход:

1. Primary path — `Electron + Python sidecar`.
2. Experimental path — `Tauri + тот же Python sidecar`.
3. Fast fallback — `pywebview + тот же Python sidecar`.

Почему именно так:

- сохраняется текущий Django UI;
- не требуется отдельный UI rewrite;
- Electron убирает runtime-зависимость от WebView2 и делает первый запуск на Windows предсказуемее;
- Tauri остается экспериментом на случай, если размер дистрибутива станет важнее.

Минимальный план:

1. Поднять локальный sidecar на `waitress`.
2. Завести Electron shell вокруг sidecar.
3. Подготовить NSIS per-user installer.
4. Проверить установку без admin-прав, Excel-выгрузки и SQLite data-dir.
5. Проверить Tauri отдельно как эксперимент, не блокирующий Electron-релиз.

### Этап E. Focused Operational Analytics

Цель: добавить полезную аналитику, но только после закрытия data-safety и пилотной установки.

Принцип: аналитика должна отвечать на простые операционные вопросы владельца, а не превращать систему в BI-комбайн.

Приоритет:

1. Что заканчивается:
   - позиции ниже минимального остатка;
   - позиции с быстрым расходом;
   - простой список "докупить".
2. Что лежит без движения:
   - slow-moving / stale positions;
   - позиции без движения за период;
   - позиции с нулевым или отрицательным движением.
3. Что двигалось активнее всего:
   - top positions by incoming/outgoing/net movement;
   - top categories by movement.
4. Простые графики по периоду:
   - динамика прихода/расхода;
   - структура остатков;
   - top positions by movement.

Не делать:

- тяжелый BI-dashboard в стиле отдельного продукта;
- аналитику, которая требует продаж/цен/маржи до подтверждения торгового контура;
- Superset до перехода на PostgreSQL и появления реальной BI-потребности.

### Этап F. Плановый контур и прогнозирование

Цель: добавить слой операционного планирования поверх фактического учета.

Идея:

- вести календарь ожидаемых поставок;
- вести календарь планируемых отгрузок / продаж;
- хранить не только факт, но и будущие обязательства по датам;
- на этой базе строить прогноз доступного остатка по складу и позиции.

Практический смысл:

- видеть ожидаемый дефицит заранее;
- понимать, в какие даты склад уйдет ниже безопасного уровня;
- оценивать влияние будущих поставок и отгрузок на доступность номенклатуры;
- готовить простой операционный forecast без полноценной ERP/MRP.

Минимальный возможный сценарий:

1. Отдельный календарь плановых событий:
   - `ожидаемое поступление`;
   - `плановая отгрузка`.
2. У каждого события есть:
   - дата;
   - склад;
   - позиция;
   - количество;
   - статус (`план`, `подтверждено`, `отменено`, `исполнено`).
3. Прогноз остатка считается как:
   - текущий расчетный остаток;
   - плюс будущие подтвержденные поступления;
   - минус будущие подтвержденные отгрузки;
   - с просмотром по дням/неделям на горизонте.

Важно:

- это отдельный будущий слой, а не часть текущего MVP;
- прогноз не должен подменять фактический учет;
- сначала должен оставаться один источник истины для факта, потом поверх него - плановый календарь и forecast.

### Этап G. Client-side index для тяжелых списков

Цель: проверить, может ли компактный клиентский индекс сделать большие списочные экраны быстрее и удобнее без перехода к полноценному SPA и без потери серверной консистентности.

Идея паттерна:

- сервер отдает компактный JSON-индекс строк списка;
- клиент локально фильтрует, сортирует и переключает страницы;
- в DOM рендерится только текущий видимый кусок, а не вся таблица;
- детали строки открываются обычным переходом на server-rendered detail page;
- полный объект, строки документа, импортные данные и другие тяжелые детали не входят в индекс.

Граница первого этапа:

- целевой рабочий предел trial: до `10 000` компактных строк;
- выше этого нужен chunking, server-side search или hybrid search;
- индекс должен быть маленьким и измеряемым: payload size, rows count, time-to-filter, render time, browser memory;
- любое внедрение должно сохранять обычный server-side fallback через текущие URL-фильтры и пагинацию.

Лучшие кандидаты:

1. Журнал документов движения.
   - Высокая ценность: поиск по номеру, типу, статусу, складу, дате, комментарию.
   - Низкий риск: детальная карточка документа остается lazy через обычный detail URL.
   - Хороший trial-screen: уже есть быстрые фильтры и понятные табличные поля.
2. Журнал инвентаризаций.
   - Средняя ценность: меньше записей, но похожая модель статусов и дат.
   - Можно внедрять после документов, если trial подтвердит пользу.
3. Номенклатура.
   - Полезно при большом справочнике.
   - Осторожно: нужно не ломать server-side import/onboarding flow и role-aware actions.

Где не использовать на первом этапе:

- текущие остатки, если расчет зависит от выбранного склада, нулевых позиций и актуальных движений прямо сейчас;
- подневный и помесячный отчеты, где объем легко растет как `дни × позиции × склады`;
- периодную аналитику, где важнее серверная консистентность агрегатов и экспорт с теми же параметрами;
- import preview, где row-level validation и commit-flow должны оставаться серверным источником истины;
- экраны с row-level permissions, если позже появятся объектные права.

Technical fit:

- текущий проект остается Django server-rendered UI, поэтому не нужно механически тащить `TanStack Table`;
- первый trial лучше делать как progressive enhancement: обычная страница работает без JS, а JS-индекс улучшает фильтрацию и сортировку при включенном браузере;
- если vanilla JS станет неудобным, можно рассмотреть маленький frontend island или table-library точечно;
- virtualization/windowing нужна только если заметны проблемы render time, а не заранее;
- server-side permissions и доступ к detail/action URL остаются обязательными независимо от клиентского фильтра.

Риски:

- устаревание индекса после проведения или создания документов;
- рост payload на слабых офисных машинах;
- browser memory на больших справочниках;
- расхождение клиентского поиска и Excel-выгрузки, если export продолжит использовать server-side query;
- ложное ощущение безопасности, если фильтрация прав делается на клиенте;
- усложнение UI без реальной пользы при текущих объемах данных.

Минимальный trial plan:

1. Выбрать только журнал документов движения.
2. Добавить server endpoint, который отдает компактный индекс только разрешенных строк:
   - `id`;
   - `number`;
   - `date` (`operation_date`);
   - `type`;
   - `status`;
   - `warehouse`;
   - `destination_warehouse`, если есть;
   - `lines_count`;
   - `comment` в усеченном виде;
   - `detail_url`.
3. Добавить feature flag или режим страницы, чтобы можно было сравнить старый и новый UX.
4. Сохранить текущие server-side фильтры, пагинацию и Excel export как fallback.
5. Измерить:
   - количество строк;
   - размер JSON payload;
   - время загрузки индекса;
   - время фильтрации по строке поиска;
   - время сортировки;
   - количество DOM-строк;
   - поведение на 1k, 5k и 10k документов.
6. После trial решить:
   - оставить только progressive enhancement;
   - расширить на инвентаризации и номенклатуру;
   - перейти на chunking/hybrid;
   - отказаться, если server-side UX достаточно хорош.

Документы, которые нужно обновить перед реализацией:

- `docs/TECH_SPEC.md`:
  - добавить спорное требование про client-side index для тяжелых списков;
  - зафиксировать, что server-side permissions и server-side fallback обязательны;
  - зафиксировать порог trial до `10 000` компактных строк.
- `docs/ARCHITECTURE.md`:
  - описать index endpoint как read-only presentation layer;
  - запретить перенос доменных правил, прав доступа и export-логики на клиент;
  - описать lazy detail pattern.
- `docs/STATUS.md`:
  - после реализации trial зафиксировать фактические метрики и решение.

## 3. SQLite и масштабирование до нескольких пользователей

### Финальная позиция

SQLite подходит для:

- разработки;
- demo mode;
- пилота;
- малого локального контура с невысокой интенсивностью записи;
- режима "один компьютер, один активный оператор".

SQLite может быть приемлем для нескольких пользователей только как отдельно проверенный сценарий, если:

- пользователи в основном смотрят данные;
- записи происходят не непрерывно, а эпизодически;
- нет постоянного одновременного проведения большого числа документов.

Важно: это не должно превращаться в публичное обещание production-grade многопользовательской работы на SQLite.

SQLite начнет мешать, если:

- несколько операторов регулярно одновременно проводят документы;
- появляются задержки или `database is locked`;
- растет число параллельных write-операций;
- нужна более предсказуемая серверная эксплуатация и резервирование.

### Практический вывод

- Для первого реального запуска `SQLite` можно оставить, если явно ограничить deployment mode.
- Supported ранний commercial mode: `Local Single User` - один локальный компьютер, один активный оператор, SQLite, local backup/restore.
- Future commercial mode: `Team / Multi-User` - тот же доменный core, но server/PostgreSQL deployment для нескольких пользователей.
- Для устойчивой многопользовательской эксплуатации стоит планировать переход на `PostgreSQL`, не создавая отдельную ветку складской бизнес-логики.
- Переход желательно делать до того, как система станет критичной для ежедневной работы.

### Почему переход на PostgreSQL здесь не должен быть болезненным

Проект построен на Django ORM, поэтому основная доменная логика не зависит от конкретной БД.

Цель миграции - сменить deployment profile, а не создать второй продукт. Расчет остатков, проведение документов, инвентаризация, import/export и audit events должны оставаться общими.

Нормальный путь миграции:

1. Поднять `PostgreSQL`.
2. Настроить `DATABASES` в Django settings.
3. Прогнать `migrate`.
4. Перенести данные.
5. Переключить приложение на новую БД.

То есть архитектурной ловушки сейчас нет, но есть технический долг по конкурентным операциям, который все равно нужно закрывать независимо от БД.

### Рекомендуемая политика

- Пока проект в стадии MVP/pilot: `SQLite`.
- Перед коммерческим пилотом: явно показать пользователю backup/restore и deployment limits.
- Перед активной многопользовательской эксплуатацией: `PostgreSQL`.
- Для BI и внешних dashboards: почти наверняка `PostgreSQL`.

## 4. Встроенная аналитика через pandas / polars

### Что здесь имеет смысл

Добавить модуль "легкой аналитики", который:

- берет те же фильтры периода, склада и представления, что уже есть в UI;
- считает дополнительные агрегаты поверх выгрузки/набора строк;
- показывает результат прямо в браузере;
- не ломает текущий UX и не создает отдельный параллельный продукт внутри системы.

### Что лучше считать через ORM, а что через pandas/polars

Через ORM удобно:

- текущие остатки;
- базовые period reports;
- фильтрацию по складу, дате, статусу;
- простые агрегаты, близкие к доменной логике.

Через `pandas` или `polars` удобно:

- pivot / cross-tab;
- группировки по нескольким измерениям;
- доли, ranking, top-N;
- более сложные табличные summary и экспериментальную аналитику.

### Выбор между pandas и polars

`pandas`:

- привычнее;
- проще для большинства разработчиков;
- лучше подходит для первого аналитического модуля.

`polars`:

- быстрее на больших таблицах;
- очень удобен для columnar-style агрегаций;
- хорош, если аналитических сценариев станет заметно больше.

### Практическая рекомендация

- Для первого встроенного аналитического модуля начать с `pandas`, если нужен быстрый и предсказуемый результат.
- Если аналитика начнет расти по объему и сложности, можно перейти на `polars` либо использовать его точечно.

## 5. Как показывать такую аналитику в браузере

### Нормальный UX-подход

Не делать отдельный "аналитический кабинет" с другой логикой фильтров.

Правильнее:

1. Оставить существующие фильтры периода/склада.
2. Добавить вкладку или блок "Базовая аналитика" рядом с текущим периодным отчетом.
3. Показывать:
   - summary cards;
   - 1-3 полезных графика;
   - компактные вспомогательные таблицы.

### Что не стоит делать для product UI

- Не делать основной веб-UI на `seaborn` картинках.
- Не тащить тяжелые notebook-style визуализации в production flow.
- Не строить много графиков без реальной операционной пользы.

### Что стоит делать

- backend считает агрегаты;
- frontend получает компактный JSON;
- графики рисуются в браузере.

Рекомендуемые варианты:

- `Plotly` — если нужна интерактивность;
- `Altair` — если нужна аккуратная декларативная визуализация;
- `Chart.js` — если нужны простые и легкие product-графики.

### Практическая рекомендация

Для этого проекта самый прагматичный путь:

- табличные расчеты в Django + ORM или `pandas/polars`;
- графики в браузере через `Chart.js` или `Plotly`;
- единые фильтры и единый экран, без UX-разрыва.

## 6. Superset

### Можно ли тянуть данные в Superset

Технически да, но как рабочую траекторию лучше рассматривать не `SQLite`, а `PostgreSQL`.

Причина проста:

- Superset естественнее работает с серверной БД;
- проще давать read-only доступ;
- проще строить SQL views и dashboards;
- проще эксплуатация и масштабирование.

### Рекомендуемый путь

1. Перевести основную БД на `PostgreSQL`.
2. Подключить Superset read-only пользователем.
3. Для BI-потребностей сделать отдельные представления или curated views:
   - движения за период;
   - остатки по складам;
   - инвентаризационные отклонения;
   - агрегаты по группам/единицам/типам движения.
4. Использовать Superset для manager-facing dashboards, а не для операционного ввода документов.

### Практический вывод

- Superset сюда хорошо ложится как внешний BI-слой.
- Но правильный момент для него — после перехода на `PostgreSQL`.
- Внутренний UI и Superset должны дополнять друг друга, а не конкурировать:
  - WMS UI = операции и базовая аналитика;
  - Superset = дашборды, сравнения, BI-срезы.

## 7. Рекомендуемая целевая траектория

Если идти прагматично, без перегруза MVP:

1. Закрыть commercial pilot readiness:
   - backup/restore;
   - user attribution;
   - audit hardening;
   - deployment limits;
   - проверка реальных Excel-файлов остается отдельным следующим slice.
2. Довести desktop packaging до надежного Windows installer flow.
3. Только после этого добавлять легкую встроенную аналитику в браузере.
4. При первых признаках реальной многопользовательской работы перейти на `PostgreSQL`.
5. После этого подключать Superset как внешний слой управленческой аналитики.

## 8. Финальные мысли

- `SQLite` сейчас не ошибка, а нормальный выбор для стадии продукта.
- `PostgreSQL` нужен не “потому что так принято”, а когда появляется многопользовательская эксплуатация и требования к устойчивости.
- Встроенная аналитика через `pandas/polars` вполне совместима с текущим UX, но она не важнее backup, attribution и installer reliability.
- `Superset` сюда хорошо вписывается, но как следующий слой зрелости, а не как первый шаг.
- Для этого проекта лучше эволюционный путь: сначала безопасный локальный пилот, затем desktop packaging, затем легкая аналитика, затем многопользовательская надежность и BI.
