# ARCHITECTURE.md

Последнее обновление: 2026-03-19

## Текущая архитектура

Приложение пока сознательно остается монолитным Django MVP.

Это означает:

- один Django app: `warehouse_app`;
- SQLite для локального контура;
- server-rendered templates без отдельного SPA-фронтенда;
- бизнес-логика сосредоточена в доменных моделях и сервисных функциях;
- отдельный `desktop/`-контур для будущей упаковки в GUI-приложение без дублирования доменной логики.

Для текущего размера проекта это нормальная точка. Архитектура еще не требует дробления на несколько apps.

## Разделение ответственности

- `models.py`
  - доменные сущности;
  - статусы и типы документов;
  - правила проведения документов;
  - правила проведения инвентаризации;
  - coarse-grained concurrency hardening через `select_for_update()` и lock по складу.
- `services.py`
  - расчет остатков;
  - построение расширенных остатков (`приход / расход / остаток`, нулевые позиции по запросу);
  - построение плоских отчетов `Дни` и `Месяцы`;
  - расчет периодной аналитики;
  - построение Excel workbook payload без знания про HTTP.
- `activity.py`
  - запись и чтение операционной истории событий;
  - idempotent event creation for posting workflows.
- `ActivityEvent`
  - хранит компактную историю складо-влияющих событий;
  - не является security audit log и не содержит user attribution.
- `UserProfile`
  - хранит простую роль пользователя: admin, operator или viewer;
  - не является workspace/account model и не реализует object-level permissions.
- `permissions.py`
  - центральные role helper-функции и decorators для server-side запрета write-действий;
  - анонимный локальный режим сознательно сохраняет текущие MVP-права для demo/desktop совместимости.
- `demo.py`
  - безопасная проверка, что база пуста;
  - загрузка демонстрационного набора данных;
  - controlled reset и повторная перезагрузка demo-набора.
- `forms.py`
  - HTML-формы;
  - formset-ы строк документов и инвентаризации.
- `views.py`
  - прием параметров из HTTP;
  - orchestration между формами, сервисами и шаблонами;
  - встроенные quick-filter presets для часто используемых списочных представлений;
  - без сложной доменной логики.
- `tests.py`
  - проверка критичных доменных сценариев.
- `version.py`
  - единый источник версии приложения для UI, документации и desktop-сборки.
- `desktop/python_sidecar`
  - локальный WSGI-launcher на `waitress`;
  - sidecar-контракт для desktop-оболочек;
  - выбор пользовательского data-dir для SQLite вне bundled app.
- `desktop/pywebview_shell`
  - быстрый fallback-shell без UI rewrite.
- `desktop/electron_shell`
  - основной Windows desktop-shell;
  - запускает Python sidecar, ждет `/healthz/`, открывает Django UI в окне.
- `desktop/tauri_shell`
  - экспериментальный desktop-shell для Windows packaging.
- `desktop/build`
  - стартовые `PyInstaller` spec-файлы и build-скрипты prototype-path.

## Почему это подхватываемо другим агентом

- есть живое ТЗ в `docs/TECH_SPEC.md`;
- есть архитектурное описание в этом файле;
- есть отдельный статус проекта в `docs/STATUS.md`;
- модульные границы простые и легко читаются.

## Ключевые инварианты

- Источник истины для остатков: проведенные строки `StockDocumentLine`.
- Инвентаризация не пишет остатки напрямую.
- Периодная аналитика строится поверх тех же движений, а не из отдельной витрины.
- UI не должен дублировать бизнес-правила, которые уже есть в моделях и сервисах.
- Demo reset допустим только в явном demo-сценарии.
- Desktop-shell не должен переносить в себя доменные правила и отчетную логику; он только поднимает локальный контур и открывает окно.
- Встроенные quick-filter presets остаются view-level настройками URL-параметров. Пользовательские сохраненные представления, если появятся позже, должны проектироваться отдельно после ролей и модели владения.
- Ролевые ограничения должны проверяться на сервере. Template-гейтинг используется только для UX и не считается защитой сам по себе.

## Когда нужно рефакторить дальше

Следующие признаки означают, что проект пора дробить сильнее:

- `views.py` становится трудно читать из-за новых отчетов и workflow.
- В `services.py` появятся несколько независимых областей логики.
- Появятся роли, API, интеграции или фоновая обработка.

Тогда имеет смысл перейти к такой структуре:

- `warehouse_app/domain/`
- `warehouse_app/reports/`
- `warehouse_app/exports/`
- `warehouse_app/web/`

Но сейчас это преждевременно.

## Правила для будущих изменений

- Новые отчеты сначала добавлять в `services.py`, потом подключать через `views.py`.
- Если вводится новая доменная операция, ее инварианты должны жить рядом с моделью.
- Если появляется тяжелая отчетность, можно вынести ее в отдельный модуль, но без дублирования правил остатков.
- HTTP-ответы для файлов должны оставаться в `views.py`, а не возвращаться напрямую из сервисов.
- Если развивается desktop/gui-app ветка, ее orchestration должна оставаться в `desktop/`, а не смешиваться с доменной частью `warehouse_app/`.
