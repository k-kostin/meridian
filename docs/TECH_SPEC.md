# TECH_SPEC.md

Живое ТЗ проекта. Это рабочий источник требований до момента, пока не появится формально утвержденное ТЗ.

Последнее обновление: 2026-06-10

## Назначение

Система складского учета для небольшого предприятия. Формулировки в интерфейсе и коде должны оставаться нейтральными, без привязки к конкретной товарной группе.

## Подтвержденные требования

- Просмотр текущих остатков по складу.
- Нужна отдельная возможность смотреть остатки и аналитику:
  - сводно по выбранным складам;
  - с разбивкой по складам.
- Редактирование номенклатуры.
- Фиксация прихода и расхода.
- Поддержка ручной корректировки.
- Поддержка инвентаризации как отдельного документа.
- При проведении инвентаризации должна автоматически создаваться корректировка на разницу.
- Инвентаризация должна уметь задавать текущее фактическое состояние склада.
- Должен существовать этап первичной настройки и ввода текущего склада.
- Первичное состояние склада должно вноситься через стартовую полную инвентаризацию.
- Должна быть выгрузка в Excel.
- В Excel-выгрузках должен быть отдельный служебный лист с параметрами выгрузки.
- Для периодной аналитики в режиме `По складам` допустимы отдельные дополнительные листы Excel по каждому складу, если основной лист остается плоским и пригодным для последующей фильтрации.
- Формат количества должен зависеть от единицы измерения.
- Должен быть demo mode для показа системы на демонстрационных данных.
- В списочных разделах и на текущих остатках нужен поиск и пагинация, пригодные для расширенной номенклатуры.
- При очистке поисковой строки в списочных разделах полный список должен возвращаться без отдельного ручного нажатия на кнопку поиска.
- Проведенные складские операции должны иметь видимую историю событий на детальных страницах документов и инвентаризаций.
- Нужен просмотр состояния склада:
  - на дату;
  - на месяц;
  - на год;
  - на произвольный период.
- Для периода длиннее одного дня отчет должен показывать:
  - состояние на начало периода;
  - состояние на конец периода;
  - дельту за период;
  - показатели движения внутри периода.
- В журнале документов движения нужен фильтр по диапазону дат, а выгрузка движений в Excel должна наследовать выбранный диапазон.
- Экран текущих остатков должен показывать не только текущий остаток, но и суммарный приход и суммарный расход по позиции.
- На текущих остатках должна быть доступна опция показа нулевых позиций, чтобы экран мог показывать весь справочник, а не только ненулевые остатки.
- Нужен отдельный плоский отчет, эквивалентный Excel-листу `Дни`: одна строка = одна позиция в один день, с полями `на начало / приход / расход / дельта / на конец`.
- Нужен отдельный плоский отчет, эквивалентный Excel-листу `Месяц`: одна строка = одна позиция в один месяц, с полями `на начало / приход / расход / дельта / на конец`.
- Нужен отдельный Windows desktop / gui-app контур, в котором пользователь не ставит Python вручную и не работает через внешний браузер или видимую командную строку.
- Приложение должно иметь явную версию, видимую в интерфейсе и согласованную с desktop package version.
- В журнале документов движения и на текущих остатках нужны быстрые встроенные фильтры для частых представлений без ручной настройки формы.
- Нужны базовые роли для пилотного использования: администратор, оператор, наблюдатель.
- Авторизованный наблюдатель может читать списки, карточки, отчеты и Excel-выгрузки, но не может создавать, редактировать, проводить или перезагружать данные.
- Авторизованный оператор может выполнять складские операции и инвентаризации, но не может редактировать справочники или перезагружать демо-данные.
- Авторизованный администратор может выполнять все действия.
- Анонимный локальный режим сохраняет текущий MVP-flow для demo/desktop совместимости.
- Импорт из Excel должен начинаться с безопасного preview/validation слоя для номенклатуры.
- Импорт должен сообщать ошибки на уровне строк до записи изменений.
- Предпросмотр импорта должен быть доступен только пользователям, которые могут управлять справочниками.
- Commit-flow импорта номенклатуры должен по умолчанию создавать только новые позиции после успешной проверки всего файла.
- Импорт номенклатуры должен поддерживать распространенные алиасы колонок без изменения доменной логики.
- Обновление существующих позиций через импорт должно быть доступно только как явный opt-in режим, а не как поведение по умолчанию.
- Автоматическое создание недостающих единиц измерения через импорт должно быть доступно только как явный opt-in режим.
- Импорт не должен обходить доменные правила складского учета.
- Импорт стартовых остатков должен создавать черновик полной инвентаризации, а не прямые записи остатков.
- Номенклатура должна поддерживать опциональные категории.
- Операционные списки и отчеты должны уметь фильтроваться по категории номенклатуры там, где это не ломает расчетную консистентность.
- Пользователь должен иметь возможность сохранять часто используемые представления для ключевых списков без замены встроенных быстрых фильтров.
- До реального коммерческого пилота система должна иметь понятный backup/restore flow для локальной базы.
- Local Single User профиль должен поддерживать ручное создание и скачивание резервной копии локальной SQLite-базы.
- Restore локальной SQLite-базы должен быть описан и доступен через management command с явным подтверждением; web restore не должен появляться без отдельного безопасного design.
- Перед автоматическими desktop migrations должен создаваться `pre_migration` backup, если локальная база уже существует.
- До реального коммерческого пилота операционная история должна фиксировать пользователя для ключевых write/posting действий.
- Локальный SQLite-контур должен явно позиционироваться как single-computer / single-active-operator deployment, пока не проверен отдельный PostgreSQL/server path.
- Коммерческие варианты поставки должны использовать одно доменное ядро и одни правила складского учета. Различия между Local Single User и Team / Multi-User профилями должны жить в конфигурации, упаковке, БД и операционных гарантиях, а не в отдельной версии бизнес-логики.
- Local Single User профиль: SQLite, один локальный компьютер, один активный оператор, понятный data-dir и backup/restore.
- Team / Multi-User профиль нельзя обещать без отдельного server/PostgreSQL deployment path, user attribution, backup policy и проверенной конкурентной эксплуатации.
- Проект не должен незаметно расширяться до ERP: функции продаж, бухгалтерии, POS, маркетплейсов, цен и поставщиков должны добавляться только после отдельного подтверждения сценария.

## Что реализовано сейчас

- Справочники:
  - единицы измерения;
  - склады;
  - номенклатура.
- Документы:
  - приход;
  - расход;
  - корректировка;
  - инвентаризация.
- На детальных страницах документов движения и инвентаризаций показывается история ключевых событий: проведение документа, проведение инвентаризации, создание автокорректировки.
- Экран первичной настройки.
- Стартовая инвентаризация.
- Расчетные остатки.
- Расширенный экран текущих остатков:
  - приход всего;
  - уход всего;
  - остаток;
  - опциональный показ нулевых позиций.
- Отдельный подневный отчет (`Дни`) с плоской таблицей `дата × позиция`.
- Отдельный помесячный отчет (`Месяцы`) с плоской таблицей `месяц × позиция`.
- Аналитика периода:
  - день;
  - месяц;
  - год;
  - произвольный период.
- Два представления остатков и периодной аналитики:
  - сводно по складам;
  - по складам.
- Excel-экспорт:
  - номенклатура;
  - остатки;
  - движения;
  - инвентаризации;
  - подневный отчет;
  - помесячный отчет;
  - периодная аналитика.
  - отдельный лист `Параметры` с метаданными выгрузки.
  - дополнительные листы по каждому складу в периодной аналитике при режиме `По складам`.
- Поиск, пагинация и выбор размера страницы:
  - в номенклатуре;
  - в документах движения;
  - в инвентаризациях;
  - в текущих остатках;
  - в справочниках единиц и складов.
- Быстрые встроенные фильтры:
  - в документах движения: черновики, проведенные, приходы, расходы, перемещения;
  - в текущих остатках: по складам, сводно, все позиции, только с остатком.
- Базовая ролевая UX-модель:
  - профиль пользователя с ролью `admin / operator / viewer`;
  - отображение роли в интерфейсе;
  - server-side запрет write-действий для ролей без права;
  - базовая страница входа через Django auth.
- Подготовлен foundation-слой импорта Excel:
  - parser-only чтение листа `Номенклатура`;
  - preview-строки для `Артикул / Наименование / Единица / Активна / Комментарий`;
  - ошибки на уровне строк для обязательных полей;
  - без записи в БД и без импорта остатков.
- Добавлен экран предпросмотра импорта номенклатуры:
  - загрузка `.xlsx`;
  - отображение строк предпросмотра;
  - отображение ошибок по строкам;
  - доступ только для роли, управляющей справочниками;
  - commit-flow для создания новых позиций при полностью валидном файле;
  - блокировка неизвестных единиц, существующих артикулов и дублей артикулов внутри файла.
- Импорт номенклатуры поддерживает:
  - распространенные алиасы колонок;
  - явный режим обновления существующих позиций;
  - явный режим автоматического создания недостающих единиц измерения;
  - сохранение прежнего create-only поведения по умолчанию.
- Категории номенклатуры:
  - отдельный справочник категорий;
  - опциональная категория у позиции;
  - фильтр по категории в списке номенклатуры.
- Фильтр по категории доступен в:
  - текущих остатках;
  - подневном отчете;
  - помесячном отчете;
  - аналитике периода;
  - соответствующих Excel-выгрузках.
- Пользовательские сохраненные представления реализованы для:
  - журнала документов движения;
  - текущих остатков.
- Добавлен импорт стартовых остатков:
  - загрузка `.xlsx` с листом `Стартовые остатки` или активным листом;
  - preview-строки по `Склад / Артикул / Фактическое количество / Комментарий`;
  - row-level validation errors для обязательных полей, складов, артикулов, дублей и количества;
  - commit-flow создает черновик полной инвентаризации для одного склада;
  - проведение созданной инвентаризации остается отдельным ручным действием пользователя.
- При очистке поисковой строки в списочных разделах интерфейс автоматически возвращает полный список без отдельного ручного сабмита.
- В документах движения доступны фильтры `Дата от` / `Дата до`, а Excel-выгрузка движений наследует выбранный диапазон дат.
- Demo mode:
  - включение через настройку окружения;
  - загрузка демонстрационных данных в пустую базу;
  - расширенный демо-набор с историей движений не менее чем за 45 дней;
  - возможность явной перезагрузки демо-набора;
  - индикация режима в интерфейсе.
- Подготовлен desktop/gui-app scaffold:
  - `desktop/python_sidecar` для локального WSGI-launcher;
  - `desktop/electron_shell` как основной Windows shell;
  - `desktop/pywebview_shell` как быстрый fallback shell;
  - `desktop/tauri_shell` как экспериментальный desktop-path.
- Версия приложения `v0.4.0` выводится в интерфейсе и синхронизирована с Electron package version.
- Product boundary после v0.4: Meridian остается lightweight local warehouse desk, а не ERP/WMS-комбайном.

## Зафиксированные бизнес-правила

- Остаток не хранится отдельной таблицей, а вычисляется по проведенным движениям.
- Непроведенные документы не влияют на остатки и аналитику.
- Расход не должен проводиться в минус.
- Полная инвентаризация должна уметь обнулять отсутствующие позиции через автоматическую корректировку.
- Периодная аналитика должна считать начало периода как остаток на день перед началом окна.
- Сводный отчет по нескольким складам должен суммировать показатели по одинаковой позиции, а не выбирать один из складов.
- Поскладской отчет должен показывать отдельную строку на сочетание `склад + позиция`.
- Demo mode может перезагружать данные только по явной команде пользователя.
- Итоги в периодной аналитике (subtotals и grand total) рассчитываются отдельно по каждой единице измерения. Смешивать количества разных единиц в одно число запрещено. Итог по складу или по всем складам — это всегда список записей, по одной на единицу измерения.
- Группировка поскладских итогов выполняется по `warehouse_id` (PK), а не по имени склада, чтобы два склада с одинаковым именем оставались отдельными группами.
- Проведенные документы движения и инвентаризации нельзя редактировать или удалять через Django admin.

## Не зафиксировано окончательно

- Нужны ли расширенные режимы импорта стартовых остатков: несколько складов в одном файле, сопоставление колонок или авто-проведение после дополнительного подтверждения.
- Нужны ли контрагенты, цены и связь со сбытом.
- Нужны ли barcode/labels и поддержка торгового оборудования.
- Нужны ли default view, sharing и расширенные права доступа для пользовательских сохраненных представлений.
- Нужны ли объектные права, приглашения пользователей и восстановление пароля.
- Stage B не должен переоткрывать уже реализованные функции: редактирование черновиков, перемещения между складами и встроенные быстрые фильтры.

## Правило обновления этого документа

- Любое новое подтвержденное требование сначала фиксируется здесь.
- Если требование спорное или не принято окончательно, оно попадает в раздел `Не зафиксировано окончательно`.
- После реализации пункта он переносится в `Что реализовано сейчас`, если раньше был только пожеланием.
