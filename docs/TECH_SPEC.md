# TECH_SPEC.md

Живое ТЗ проекта. Это рабочий источник требований до момента, пока не появится формально утвержденное ТЗ.

Последнее обновление: 2026-03-19

## Назначение

Система складского учета для небольшого предприятия. Формулировки в интерфейсе и коде должны оставаться нейтральными, без привязки к конкретной товарной группе.

## Подтвержденные требования

- Просмотр текущих остатков по складу.
- Нужна отдельная возможность смотреть остатки и аналитику:
  - сводно по выбранным складам;
  - с разбивкой по складам.
- Редактирование номенклатуры.
- Фиксация прихода и расхода.
- Поддержка ручной корректировки.
- Поддержка инвентаризации как отдельного документа.
- При проведении инвентаризации должна автоматически создаваться корректировка на разницу.
- Инвентаризация должна уметь задавать текущее фактическое состояние склада.
- Должен существовать этап первичной настройки и ввода текущего склада.
- Первичное состояние склада должно вноситься через стартовую полную инвентаризацию.
- Должна быть выгрузка в Excel.
- В Excel-выгрузках должен быть отдельный служебный лист с параметрами выгрузки.
- Для периодной аналитики в режиме `По складам` допустимы отдельные дополнительные листы Excel по каждому складу, если основной лист остается плоским и пригодным для последующей фильтрации.
- Формат количества должен зависеть от единицы измерения.
- Должен быть demo mode для показа системы на демонстрационных данных.
- В списочных разделах и на текущих остатках нужен поиск и пагинация, пригодные для расширенной номенклатуры.
- При очистке поисковой строки в списочных разделах полный список должен возвращаться без отдельного ручного нажатия на кнопку поиска.
- Проведенные складские операции должны иметь видимую историю событий на детальных страницах документов и инвентаризаций.
- Нужен просмотр состояния склада:
  - на дату;
  - на месяц;
  - на год;
  - на произвольный период.
- Для периода длиннее одного дня отчет должен показывать:
  - состояние на начало периода;
  - состояние на конец периода;
  - дельту за период;
  - показатели движения внутри периода.
- В журнале документов движения нужен фильтр по диапазону дат, а выгрузка движений в Excel должна наследовать выбранный диапазон.
- Экран текущих остатков должен показывать не только текущий остаток, но и суммарный приход и суммарный расход по позиции.
- На текущих остатках должна быть доступна опция показа нулевых позиций, чтобы экран мог показывать весь справочник, а не только ненулевые остатки.
- Нужен отдельный плоский отчет, эквивалентный Excel-листу `Дни`: одна строка = одна позиция в один день, с полями `на начало / приход / расход / дельта / на конец`.
- Нужен отдельный плоский отчет, эквивалентный Excel-листу `Месяц`: одна строка = одна позиция в один месяц, с полями `на начало / приход / расход / дельта / на конец`.
- Нужен отдельный Windows desktop / gui-app контур, в котором пользователь не ставит Python вручную и не работает через внешний браузер или видимую командную строку.
- Приложение должно иметь явную версию, видимую в интерфейсе и согласованную с desktop package version.
- В журнале документов движения и на текущих остатках нужны быстрые встроенные фильтры для частых представлений без ручной настройки формы.

## Что реализовано сейчас

- Справочники:
  - единицы измерения;
  - склады;
  - номенклатура.
- Документы:
  - приход;
  - расход;
  - корректировка;
  - инвентаризация.
- На детальных страницах документов движения и инвентаризаций показывается история ключевых событий: проведение документа, проведение инвентаризации, создание автокорректировки.
- Экран первичной настройки.
- Стартовая инвентаризация.
- Расчетные остатки.
- Расширенный экран текущих остатков:
  - приход всего;
  - уход всего;
  - остаток;
  - опциональный показ нулевых позиций.
- Отдельный подневный отчет (`Дни`) с плоской таблицей `дата × позиция`.
- Отдельный помесячный отчет (`Месяцы`) с плоской таблицей `месяц × позиция`.
- Аналитика периода:
  - день;
  - месяц;
  - год;
  - произвольный период.
- Два представления остатков и периодной аналитики:
  - сводно по складам;
  - по складам.
- Excel-экспорт:
  - номенклатура;
  - остатки;
  - движения;
  - инвентаризации;
  - подневный отчет;
  - помесячный отчет;
  - периодная аналитика.
  - отдельный лист `Параметры` с метаданными выгрузки.
  - дополнительные листы по каждому складу в периодной аналитике при режиме `По складам`.
- Поиск, пагинация и выбор размера страницы:
  - в номенклатуре;
  - в документах движения;
  - в инвентаризациях;
  - в текущих остатках;
  - в справочниках единиц и складов.
- Быстрые встроенные фильтры:
  - в документах движения: черновики, проведенные, приходы, расходы, перемещения;
  - в текущих остатках: по складам, сводно, все позиции, только с остатком.
- При очистке поисковой строки в списочных разделах интерфейс автоматически возвращает полный список без отдельного ручного сабмита.
- В документах движения доступны фильтры `Дата от` / `Дата до`, а Excel-выгрузка движений наследует выбранный диапазон дат.
- Demo mode:
  - включение через настройку окружения;
  - загрузка демонстрационных данных в пустую базу;
  - расширенный демо-набор с историей движений не менее чем за 45 дней;
  - возможность явной перезагрузки демо-набора;
  - индикация режима в интерфейсе.
- Подготовлен desktop/gui-app scaffold:
  - `desktop/python_sidecar` для локального WSGI-launcher;
  - `desktop/electron_shell` как основной Windows shell;
  - `desktop/pywebview_shell` как быстрый fallback shell;
  - `desktop/tauri_shell` как экспериментальный desktop-path.
- Версия приложения `v0.2.0` выводится в интерфейсе и синхронизирована с Electron package version.

## Зафиксированные бизнес-правила

- Остаток не хранится отдельной таблицей, а вычисляется по проведенным движениям.
- Непроведенные документы не влияют на остатки и аналитику.
- Расход не должен проводиться в минус.
- Полная инвентаризация должна уметь обнулять отсутствующие позиции через автоматическую корректировку.
- Периодная аналитика должна считать начало периода как остаток на день перед началом окна.
- Сводный отчет по нескольким складам должен суммировать показатели по одинаковой позиции, а не выбирать один из складов.
- Поскладской отчет должен показывать отдельную строку на сочетание `склад + позиция`.
- Demo mode может перезагружать данные только по явной команде пользователя.
- Итоги в периодной аналитике (subtotals и grand total) рассчитываются отдельно по каждой единице измерения. Смешивать количества разных единиц в одно число запрещено. Итог по складу или по всем складам — это всегда список записей, по одной на единицу измерения.
- Группировка поскладских итогов выполняется по `warehouse_id` (PK), а не по имени склада, чтобы два склада с одинаковым именем оставались отдельными группами.

## Не зафиксировано окончательно

- Нужны ли роли `admin / operator / viewer`.
- Нужен ли импорт из Excel для номенклатуры и стартовых остатков.
- Нужны ли перемещения между складами отдельным типом документа.
- Нужны ли категории номенклатуры.
- Нужны ли контрагенты, цены и связь со сбытом.
- Нужно ли редактирование черновиков после создания через отдельные экраны.
- Нужно ли запретить редактирование проведенных документов на уровне admin-интерфейса.
- Нужны ли пользовательские сохраненные представления с именами, владельцем, правами доступа и выбором представления по умолчанию.
