# Desktop / GUI App

Эта папка фиксирует новую ветку разработки: упаковка текущего Django-приложения в desktop-формат, прежде всего под Windows.

## Выбранные траектории

### 1. Primary path: Electron shell + Python sidecar

Это основной путь для customer-facing Windows-приложения.

Почему он выбран:

- не требует переписывать доменную логику и HTML-экраны на Qt/Flet;
- дает нативное окно без адресной строки и вкладок;
- несет Chromium с собой и не зависит от установленного WebView2;
- предсказуемее для первого запуска без admin-прав и без интернета;
- Python-часть можно упаковать отдельно как sidecar.

В этой траектории:

- Django остается источником UI и бизнес-логики;
- Python sidecar поднимает локальный WSGI-сервер только на `127.0.0.1`;
- Electron управляет окном, lifecycle sidecar, упаковкой и установщиком.

### 2. Experimental path: Tauri shell + Python sidecar

Tauri оставлен как экспериментальный путь.

Когда к нему есть смысл возвращаться:

- размер Electron-дистрибутива станет реальной проблемой;
- WebView2 availability будет подтверждена на целевых машинах;
- появится время на отдельную Windows validation matrix.

### 3. Fast fallback: pywebview shell + тот же Python sidecar

Это быстрый резервный путь для раннего desktop-прототипа и внутренней сборки.

Почему он оставлен:

- почти не требует нового стека;
- можно быстро доказать жизнеспособность desktop-сценария;
- использует тот же sidecar-контракт, что и Tauri;
- полезен как промежуточный этап до полноценной Tauri-сборки.

Минус: итоговый бандл будет тяжелее и менее "чистым" с точки зрения долгосрочного desktop-продукта.

Практический нюанс:

- на Windows этот путь чувствителен к версии Python;
- `pywebview` может тянуть `pythonnet`;
- для раннего prototype-path безопаснее ориентироваться на Python `3.12` или `3.13`;
- `sidecar` при этом можно собирать и проверять отдельно уже сейчас.

## Что сознательно не выбрано сейчас

- `PySide6 / Qt` — сильный вариант, но требует переписывать текущий UI.
- `Flet` — быстрее, чем Qt, но все равно означает новый UI-контур.

## Структура

- `python_sidecar/` — локальный WSGI-launcher и правила sidecar-контракта.
- `electron_shell/` — основной Windows shell на Electron.
- `pywebview_shell/` — быстрый Python-only shell для раннего desktop-прототипа.
- `tauri_shell/` — экспериментальная desktop-оболочка.
- `build/` — packaging-заметки и стартовые build-скрипты.

## Общий desktop-контракт

Для обеих траекторий интерфейс должен вести себя так:

- пользователь не ставит Python отдельно;
- пользователь не видит командную строку;
- пользователь не работает с внешним браузером;
- приложение открывается как обычное окно;
- backend слушает только `localhost`;
- доменная логика остается в текущем Django-контуре, а не дублируется в shell.

## Ближайшие шаги

1. Довести `python_sidecar` до стабильного локального launcher-контура.
2. Собрать и проверить `Electron` shell для Windows.
3. Подготовить NSIS per-user installer.
4. Оставить `Tauri` как отдельный эксперимент, не блокирующий релиз.
